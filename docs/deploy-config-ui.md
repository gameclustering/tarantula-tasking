# Deploy Config UI — Design

**Date:** 2026-06-28

## Goal

Replace hand-authored `deploy.json` files in the `tarantula-deploy` repo with a form in the admin UI that generates and commits them automatically.

---

## Current Flow

1. Developer hand-edits `<platform>/<appName>/deploy.json` in `tarantula-deploy` repo.
2. At task runtime, `cloud.LoadDeployConfig` clones the repo and reads the file.
3. The deploy pipeline uses `DeployConfig.Resolve(env, phase)` to merge env-specific overrides onto the `default` entry.

### Schema summary (`internal/core/deploy_config.go`)

```
DeployConfig
└── map[env]DeployEnvConfig         ("default", "dev", "stg", "prod", ...)
    ├── Steps   []string            ordered pipeline (check/create/update/build/deploy/test)
    ├── Build   PhaseConfig
    ├── Deploy  PhaseConfig
    └── Test    PhaseConfig

PhaseConfig
├── Prefix          string          short instance name prefix (e.g. "smp", "tarantula")
├── InstanceNumber  int             how many VM instances for this phase/env
├── SshUser         string          SSH user on target VMs
├── BuildHost       string          single pre-existing build server
├── BuildHosts      []string        multiple build servers for fan-out
├── VaultHost       string          public Vault URL for deployed containers
├── Description     string
├── Services        []GcpServiceConfig   Docker image + network + httpBinding
├── Ports           []string        host:container port mappings (single-service apps)
├── Settings        map[string]string   platform-specific (zone, machineType, imageType, ...)
├── Credentials     *CredentialSpec     optional: auto-seed Vault secrets on first deploy
├── TestRepo        string          repo containing k6 test scripts
├── AppPrefix       string          prefix of app instances to test against
└── Promotion       *PromotionSpec  git tag to push on test pass
```

### Existing deploy.json examples

`gcp/deploy.json` — core tarantula cluster (postoffice + admin + cloud workers)

`gcp/tarantula-sample-app/deploy.json` — single-service app with test + promotion

## Proposed UI Flow

### Step 1 — Basic info

| Field | Notes |
|---|---|
| App name | used as path: `<platform>/<appName>/deploy.json` |
| Platform | `gcp` or `vultr` (select) |
| Prefix | short VM name prefix |
| SSH user | default `tarantula` |
| Vault host | default `https://vault.gameclustering.com` |
| Build host | pre-existing build server hostname |
| Build hosts | comma-separated for fan-out builds |

### Step 2 — Services

Add/remove rows per service:

| Field | Notes |
|---|---|
| Image name | e.g. `tarantula.postoffice` |
| Network | `host` or `bridge` |
| HTTP binding | e.g. `:8081` (optional) |
| Ports | `host:container` mappings (optional, for bridge mode) |

### Step 3 — Env instance counts

| Env | Instance count |
|---|---|
| default | |
| dev | |
| stg | |
| prod | |

Platform settings (zone, machineType, imageType for GCP) default to the standard values and are editable as key-value pairs.

## Backend

New endpoint: `POST /admin/deploy/config`

Request body:
```json
{
  "appName": "tarantula-sample-app",
  "platform": "gcp",
  "prefix": "smp",
  "sshUser": "tarantula",
  "vaultHost": "https://vault.gameclustering.com",
  "buildHost": "build.gameclustering.com",
  "services": [
    { "name": "tarantula.app", "network": "host", "httpBinding": ":8080" }
  ],
  "envCounts": { "dev": 1, "stg": 2, "prod": 3 },
  "settings": { "zone": "us-central1-f", "machineType": "e2-small", "imageType": "projects/debian-cloud/global/images/family/debian-12" }
}
```

Handler steps:
1. Load vault git key (`gitKey.Git.Key`, `gitKey.Git.Org`).
2. Clone/update `tarantula-deploy` repo via `util.GitClient`.
3. Construct `map[string]DeployEnvConfig` — `default` entry has the full config; per-env entries only override `InstanceNumber`.
4. Marshal to indented JSON and write to `<platform>/<appName>/deploy.json`.
5. Commit and push via git (`git add`, `git commit`, `git push`).
6. Return `{ successful: true, path: "gcp/tarantula-sample-app/deploy.json", commit: "<sha>" }`.

## What stays manual (out of scope for now)

- `credentials` spec — infrequent, complex, safer to hand-edit
- `testRepo` / `promotion` — only needed for apps with automated test phases
- `steps` override — rarely changed from the default pipeline
- Editing an existing config — read-back and pre-fill is a follow-up; initial version is create-only
